# Frontend data contract

The next-generation frontend receives one versioned export generated by `Export-GameDataToJson`. The same payload is
written to `frontend/resources/data.json` and assigned to `window.gamingGaidenData` in `data.js` for browsers that
block `fetch()` on `file:///` pages.

## Compatibility rules

- `schema_version` is currently `1`.
- Exports without `schema_version` are interpreted as version 1 for compatibility with pre-versioned installations.
- Unknown schema versions are rejected with a visible loading error instead of being rendered unpredictably.
- The four top-level collections are required: `games`, `session_history`, `daily_playtime`, and `gaming_pcs`.
- Invalid collection entries are ignored and reported as console warnings. A malformed top-level payload is rejected.
- Collections are exported in a deterministic order. Components also sort the fields they depend on before applying
  limits, so recent-session and recent-day views do not rely on SQLite's incidental row order.
- `data.js` is the primary local-file transport. `data.json` remains the HTTP/development fallback and a readable
  export artifact.

The TypeScript contract lives in `frontend/src/types/GameData.ts`. Runtime validation and normalization live in
`frontend/src/data/GameDataValidator.ts`; all data must cross that boundary before reaching a component.

## Version 1 payload

```text
GameData
  schema_version: 1
  games: Game[]
  session_history: Session[]
  daily_playtime: DailyPlaytime[]
  gaming_pcs: GamingPC[]
  export_date?: string
  hash?: string
```

Core entry requirements:

- A game requires a non-empty `name`; missing numeric totals normalize to zero.
- A session requires `game_name`, `start_time`, and numeric `duration`.
- Daily playtime requires `play_date` and numeric `play_time`.
- A gaming PC requires a non-empty `name`; a missing `in_use` value normalizes to `FALSE`.

Adding optional fields does not require a schema bump. Removing or renaming fields, changing their meaning, or making
previously valid data invalid requires a new schema version and an explicit compatibility path in the validator.
